通过 Harness Engineering 提升 Deep Agents 性能
TLDR：我们的编码 Agent 在 Terminal Bench 2.0 上从 Top 30 提升到了 Top 5。我们只改了 harness。下面是我们做 harness engineering 的方法（先剧透：自验证和 tracing 很关键）。

Harness Engineering 的目标
Harness 的目标，是把模型那种“峰值很高但波动也很大”的智能，塑造成更适合我们目标任务的执行能力。Harness Engineering 关注的是系统层：你围绕模型构建一整套工具链，用来优化任务完成率、token 效率、延迟等目标。可调设计项包括系统提示词、工具选择和执行流程。

那到底该怎么改 harness，才能真正提升 agent？

在 LangChain，我们用 Traces 来规模化理解 agent 的失败模式。如今的模型在很大程度上仍是黑盒，其内部机制不易解释；但我们可以观察文本空间里的输入输出，并把这些信号放进改进闭环。

我们用一套简单方法，迭代改进 deepagents-cli（我们的编码 agent），在 Terminal Bench 2.0 上把分数从 52.8 提升到 66.5，提升了 13.7 分。整个过程中我们只调整 harness，模型保持不变，仍是 gpt-5.2-codex。






实验设置与 Harness 的可调旋钮
我们使用 Terminal Bench 2.0（当前用于评估 agentic coding 的标准基准之一）。它包含 89 个任务，覆盖机器学习、调试、生物等多个领域。我们用 Harbor 来编排实验：它会拉起沙箱（Daytona）、驱动我们的 agent loop，并执行验证与打分。

每一步 agent 动作都会存入 LangSmith；同时记录延迟、token 数量和成本等指标。

我们能调哪些旋钮
一个 agent harness 有很多旋钮：系统提示词、工具、hooks/middleware、skills、子 agent 委派、记忆系统等。我们有意压缩优化空间，只聚焦三个维度：System Prompt、Tools 和 Middleware（我们对模型调用和工具调用周边 hook 的称呼）。

我们从默认提示词和标准工具+middleware 起步。在 GPT-5.2-Codex 下起始分数是 52.8%。这个分数并不差，今天大致在榜单 Top 30 之外一点，但还有明显提升空间。






Trace Analyzer Skill
我们希望 trace 分析流程可复用，于是把它做成了一个 Agent Skill。它是我们“跨多次运行分析错误并改进 harness”的固定配方，流程如下：

从 LangSmith 拉取实验 traces。
启动并行错误分析 agents，再由主 agent 汇总结论与建议。
聚合反馈，并对 harness 做有针对性的修改。
这种方法类似 boosting：重点盯住上一轮的错误样本。第 3 步里，人类参与（虽然不是必须）通常很有价值，用于核验并讨论候选修改。对单任务过拟合会损害泛化，还可能在其他任务上引入回归。

自动化 trace 分析节省了大量时间，也让我们能快速试验。我们很快会发布这个 skill；目前它正在用于更通用的 prompt 优化测试。






到底哪些改动真正提升了 Agent 性能
自动化 trace 分析让我们能更快 定位 agent 的失败点。常见问题包括：推理错误、未遵循任务指令、缺少测试与验证、超时等。下面按改进点展开。

Build & Self-Verify
当下模型本质上是很强的“自改进机器”。

自验证让 agent 能在单次运行内借助反馈自我改进。但模型并不会天然进入这种 build-verify 循环。

最常见失败模式是：agent 写完方案，回看一遍代码，自认为没问题，然后就停了。测试是自治编码 agent 的关键环节：它既验证整体正确性，也给 agent 提供可用于爬坡优化的反馈信号。

我们在系统提示词中补充了问题求解流程：

Planning & Discovery： 阅读任务、扫描代码库，基于任务规格和验证方式制定初始计划。
Build： 按“可验证”目标实现方案；若没有测试则补测试，覆盖 happy path 与边界场景。
Verify： 运行测试并完整阅读输出，对照的是任务要求，而不是你自己写的代码。
Fix： 分析错误，回到原始规格，修复问题。
我们非常强调测试，因为它驱动了每次迭代中的改动方向。我们发现，除了提示词，确定性的上下文注入也能明显提升验证质量。我们使用 PreCompletionChecklistMiddleware：在 agent 准备退出前拦截它，并提醒对照任务规格完成一次验证。这类似 Ralph Wiggum Loop：通过 hook 在退出点强制 agent 继续执行；我们的用途是强制验证。






给 Agent 注入环境上下文
Harness engineering 的重要部分，是把 context engineering 做成可靠的上下文交付机制。Terminal Bench 任务自带目录结构、内置工具和严格超时。

目录上下文与工具发现：LocalContextMiddleware 在 agent 启动时扫描 cwd 及父/子目录。我们还会运行 bash 命令定位 Python 等工具。上下文探索和搜索本身容易出错，提前注入上下文能缩小错误面，帮助 agent 更快“完成环境入场”。
让 Agent 写出可测代码： Agent 默认并不知道代码必须如何“可测试”。我们会明确提示：其输出将接受程序化测试（类似代码提交时的 CI 约束）。例如规格里提到的文件路径必须严格匹配，否则自动评分阶段会失败。强调边界用例能避免只覆盖 happy path。强制模型遵循测试标准，是抑制长期“slop 累积”的有效策略。
时间预算： 我们注入时间预算提醒，推动 agent 及时收敛并切换到验证阶段。Agent 往往不擅长时间估计，这个启发式在该基准环境很有帮助。现实开发未必都有严格时限，但如果不注入约束知识，agent 往往无法在时限内完成。
Agent 对环境、约束和评估标准了解越充分，就越能自治地把工作做完。

Harness 工程师的职责，是准备并交付上下文，让 agent 能自治地完成任务。

鼓励 Agent 停下来重新审视计划
一旦 agent 锁定某条路径，往往会变得短视，形成“doom loop”：在同一条错误路线做大量微小变体（某些 trace 里超过 10 次）。

我们使用 LoopDetectionMiddleware，通过工具调用 hooks 统计每个文件的编辑次数。当同一文件编辑达到阈值 N 时，会注入类似“考虑重审当前方案”的上下文。这样有时能帮助 agent 跳出死循环；当然，如果模型仍然坚信当前路径正确，也可能继续原路线。

需要强调：这属于针对当前模型局限的工程性启发式。随着模型能力提升，这类护栏大概率会逐步不再需要；但在今天，它确实能提升 agent 的正确率和自治执行能力。

如何决定在推理上花多少算力
推理模型可以连续自治运行数小时，所以我们必须决定每个子任务要投入多少推理算力。你可以在所有任务上都开最大推理预算，但多数场景更适合优化“推理算力投入”。

Terminal Bench 的超时限制带来明显权衡：更多推理有助于每步判断，但也可能消耗超过 2x 的 token/时间。gpt-5.2-codex 有 4 档推理模式：low、medium、high、xhigh。

我们的观察是：高推理预算在规划阶段很有价值，因为部分 Terminal Bench 任务非常难，好的计划能更快通向可用解。后期验证阶段也受益于更高推理预算，有助于发现错误并提交最终解。基于这个启发式，我们采用 xhigh-high-xhigh 的“reasoning sandwich”作为基线。






在规划与验证阶段投入更多推理算力。

全程只用 xhigh 反而表现较差（53.9%），主要是更容易超时；而 high 达到 63.6%。不同推理预算拆分在多次试验里差异不算很大，所以我们继续沿用该策略，最终把分数推到 66.5%。

模型更自然的方向是 Adaptive Reasoning：例如 Claude 和 Gemini 都在探索由模型自行决定推理开销。

在多模型 harness 中，一种平衡算力预算的做法是：用大模型负责规划，再 handoff 给小模型执行实现。

构建 Agent Harness 的实践要点
Agent 的设计空间很大。下面是我们在实验和 deepagents 实践中沉淀出的通用原则：

代表 Agent 做好 Context Engineering。 现在 agent 在陌生环境里的上下文组装仍然困难。提前注入目录结构、可用工具、编码最佳实践和问题求解策略，能降低因搜索不佳和可避免失误带来的错误面。
帮助 Agent 自证正确。 模型会偏向第一个“看起来可行”的解。要强力提示它通过运行测试来验证并迭代方案。对“人不在环”的自治编码系统尤其关键。
把 Tracing 当作反馈信号。 Trace 让 agent 能自评与自调。调优时要同时看工具和推理（例如模型走错路，可能是缺工具，也可能是缺指令）。
短期内识别并修复坏模式。 今天的模型并不完美。Harness 设计者的工作，是围绕当下短板做工程补偿，同时为更强模型预留演进空间。盲目重试和不验证就是典型问题。长期看这些护栏会消失，但在现阶段它们是构建稳健 agent 应用的实用手段。
Harness 要按模型做定制。Codex 与 Claude 的提示词指南都说明，不同模型需要不同提示策略。我们用早期 harness 版本测试 Claude Opus 4.6 得到 59.6%，有竞争力但仍低于 Codex，因为没有对 Claude 做同样的改进闭环。很多原则可迁移（如高质量上下文准备与验证优先），但针对你的任务做几轮 harness 迭代，通常才能把跨任务表现最大化。
Harness 设计仍有大量开放研究可做。值得探索的方向包括：多模型系统（Codex + Gemini + Claude 协同）、用于持续学习的记忆原语（让 agent 能在任务间自治进化）、以及跨模型衡量 harness 变更效果的方法。

在“外环”改进方向上，我们也在关注 RLMs，希望更高效地从 traces 中挖掘改进信号。我们会继续迭代 harness，并公开分享研究。

我们也开放了 Trace 数据集，方便社区复现与进一步分析。
