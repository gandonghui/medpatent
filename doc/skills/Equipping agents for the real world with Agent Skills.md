一、文章概述
《Equipping agents for the real world with Agent Skills》是 Anthropic 于 2025 年 10 月 16 日发布的技术文章，核心介绍了Agent Skills这一创新概念 —— 一种用文件 / 文件夹封装专业知识、让通用 AI Agent 快速具备领域能力的模块化方案。文章指出，虽然通用 Agent（如 Claude）在跨领域完成复杂任务方面表现出色，但真实工作场景需要流程化知识、组织上下文与确定性执行，仅靠大模型原生能力存在不足。

传统解决方案如为每个场景定制 Agent 或内嵌长 Prompt，导致系统碎片化、难复用、难扩展、难维护的问题。Agent Skills 的提出旨在打造一种可组合、可扩展、可移植的方式，让通用 Agent 按需加载专业能力，快速变身专用 Agent，从而解决通用模型在真实场景中的痛点。

二、Agent Skills 的核心概念与定义
2.1 Agent Skills 的本质
Agent Skills 本质上是一个结构化文件夹，内含指令、脚本、资源等，Agent 可动态发现、加载、执行，按需扩展能力。这一概念的设计灵感来源于为新员工准备的入职指南 —— 一次封装、多 Agent 复用，无需重复定制。

与传统的 Agent 定制化方案不同，Agent Skills 采用了标准化的封装格式，已成为跨平台通用能力封装规范，不局限于 Claude 生态。这种标准化设计使得技能可以在不同的 Agent 实现之间共享和复用，大大提高了开发效率。

2.2 技能的结构设计
一个完整的 Skill 是包含SKILL.md文件的目录，采用三级渐进式披露设计，以最小化上下文窗口占用：

元数据层（YAML 前置）：SKILL.md 开头必须有name和description，Agent 启动时预加载到系统 Prompt，仅用于判断是否触发 Skill（约 100 token）。
核心指令层：Agent 判定任务匹配时，调用load_skill读取完整 SKILL.md（建议小于 5000 token），获取详细操作流程。
资源层：SKILL.md 可引用额外文件（如reference.md、forms.md、Python 脚本），Agent 仅在需要时读取，实现无限上下文扩展。
这种渐进式披露的设计原理类似一本组织良好的手册，从目录开始，然后是具体章节，最后是详细附录，让 Claude 只在需要时加载信息。

2.3 技能的技术规格
根据文章定义，一个 Skill 必须满足四个标准：

程序性内容：包含操作指南（程序、工作流程、标准操作程序），而非事实检索
任务类适用性：适用于一类问题，而非单个实例
结构化组件：包括 SKILL.md 文件以及可选资源（脚本、模板、示例）
可移植性：Skills 仅基于文件系统，易于编辑、版本控制、共享，并可在不同的兼容 Skill 的 Agent 框架中使用
三、Agent Skills 的核心优势与功能
3.1 代码执行与确定性
Skill 的一个关键能力是可嵌入预写脚本（Python/Bash），Agent 按需调用执行，无需将代码 / 数据全量加载到上下文。这一特性解决了大模型在某些操作上的效率和可靠性问题。

文章指出，大语言模型在许多任务上表现出色，但某些操作更适合传统代码执行。例如，通过 token 生成进行列表排序比简单运行排序算法昂贵得多。除了效率问题，许多应用还需要只有代码才能提供的确定性可靠性。

在 PDF 技能的例子中，技能包含一个预编写的 Python 脚本，用于读取 PDF 并提取所有表单字段。Claude 可以运行这个脚本而无需将脚本或 PDF 加载到上下文中。由于代码是确定性的，这个工作流程是一致且可重复的。

3.2 模块化与组合性
Agent Skills 具有以下核心优势：

可组合性：Skills 可以堆叠在一起。Claude 自动识别需要哪些技能并协调它们的使用
可移植性：Skills 在任何地方使用相同的格式。一次构建，可在 Claude 应用、Claude Code 和 API 中使用
高效性：只在需要时加载需要的内容
强大性：Skills 可以包含可执行代码，用于传统编程比 token 生成更可靠的任务
3.3 跨平台支持
Agent Skills 已成为跨平台通用能力封装规范，不局限于 Claude 生态。目前已支持的平台包括：

Claude.ai
Claude Code
Claude Agent SDK
开发者平台
四、Agent Skills 的工作架构
4.1 Agent 四层工作架构
配合 Skills 的使用，Agent 采用四层工作架构：

Agent Loop（推理循环）：理解需求、规划行动、决策（大脑）
Agent Runtime（运行时）：执行代码、调用工具、读写文件（手脚）
MCP Servers（模型上下文协议）：连接外部系统（数据库、API、Slack 等，桥梁）
Skills Library（技能库）：领域专业知识模块，按需加载（能力库）
4.2 技能发现与加载机制
Agent Framework 包含一个技能提供器，它从文件系统目录中发现技能，并作为上下文提供器向 Agent 提供技能。它递归搜索配置路径（最多两层深）中的 SKILL.md 文件，验证其格式和资源，并向 Agent 暴露两个工具：load_skill和read_skill_resource。

技能的触发过程如下：

上下文窗口包含核心系统提示和每个已安装技能的元数据，以及用户的初始消息
Claude 通过调用 Bash 工具读取pdf/SKILL.md的内容来触发 PDF 技能
Claude 选择读取技能捆绑的forms.md文件
最后，Claude 在加载了 PDF 技能的相关指令后继续处理用户的任务
4.3 渐进式披露机制
Agent Skills 使用三阶段渐进式披露模式来最小化上下文使用：

广告阶段（每个技能约 100 token）：技能名称和描述在每次运行开始时注入系统提示，因此 Agent 知道有哪些技能可用
加载阶段（建议小于 5000 token）：当任务匹配技能的领域时，Agent 调用load_skill工具检索完整的 SKILL.md 主体和详细指令
读取资源阶段（按需）：Agent 调用read_skill_resource工具仅在需要时获取补充文件（引用、模板、资产）
五、开发与评估最佳实践
5.1 从评估起步的开发策略
文章提出了几个关键的开发和评估指南：

从评估起步：通过在代表性任务上运行 Agent 并观察它们在哪里遇到困难或需要额外上下文，来识别 Agent 能力的具体差距。然后逐步构建技能来解决这些不足。

规模化结构设计：当 SKILL.md 文件变得庞大时，将其内容拆分为单独的文件并引用它们。如果某些上下文是互斥的或很少一起使用，保持路径分离将减少 token 使用。最后，代码既可以作为可执行工具，也可以作为文档。应该清楚 Claude 是应该直接运行脚本还是将其作为参考读入上下文。

5.2 以 Agent 视角进行设计
从 Claude 的角度思考：监控 Claude 在真实场景中如何使用你的技能，并基于观察进行迭代：观察意外的轨迹或对某些上下文的过度依赖。特别注意技能的name和description。Claude 在决定是否针对当前任务触发技能时会使用这些信息。

与 Claude 协作迭代：当你与 Claude 一起处理任务时，让 Claude 捕获其成功方法和常见错误，并将其转化为技能中的可重用上下文和代码。如果在使用技能完成任务时偏离轨道，让它自我反思哪里出了问题。这个过程将帮助你发现 Claude 实际需要什么上下文，而不是试图预先预测。

5.3 技能开发的具体实践
在技能开发中，常见的模式是运行验证器、修复错误、重复的循环。例如，在文档编辑过程中：

对word/document.xml进行编辑
立即验证：python scripts/validate.py unpacked_dir/
根据验证结果进行相应处理
5.4 技能的结构设计
一个完整的技能目录结构示例如下：

expense-report/
├── SKILL.md # 必需 — 前置内容 + 指令
├── scripts/
│ └── validate.py # 可执行代码，Agent可以运行
├── references/
│ └── POLICY_FAQ.md # 按需加载的参考文档
└── assets/
└── expense-report-template.md # 模板和静态资源

SKILL.md 文件必须包含 YAML 前置内容，然后是 Markdown 内容：

---
name: expense-report
description: File and validate employee expense reports according to company policy. Use when asked about expense submissions, reimbursement rules, or spending limits.
license: Apache-2.0
compatibility: Requires python3
metadata:
author: contoso-finance
version: "2.1"
---

六、安全考虑与风险管控
6.1 安全风险分析
文章明确指出，技能通过指令和代码为 Claude 提供新能力。虽然这使它们功能强大，但也意味着恶意技能可能在使用环境中引入漏洞，或指示 Claude 泄露数据和采取意外行动。

6.2 安全最佳实践
文章提出了以下安全建议：

来源信任：只从可信来源安装技能。从不太可信的来源安装技能时，使用前要彻底审计。首先阅读技能中捆绑文件的内容，了解其功能，特别注意代码依赖项和捆绑资源如图像或脚本。

内容审计：同样，注意技能中指示 Claude 连接到潜在不可信外部网络源的指令或代码。

权限控制：将文件系统、网络和系统级访问限制为技能仅需要的内容。在执行潜在敏感操作前需要明确的用户确认。

审计与日志：记录加载了哪些技能、读取了哪些资源、执行了哪些脚本。这为你提供了一个审计跟踪，以便在出现问题时将 Agent 行为追溯到特定的技能内容。

6.3 安全架构原则
在安全架构设计上，应遵循以下原则：

最小权限原则：Agent 仅能访问完成任务所必需的资源
环境隔离：工具调用在独立容器或虚拟环境中执行
输入 / 输出过滤：对传入工具的参数和返回结果进行安全校验

八、技术评估与性能分析
8.1 SkillsBench 基准测试结果
根据 arXiv 论文《SkillsBench: Benchmarking How Well Agent Skills Work Across Diverse Tasks》的研究结果，在 84 个任务、7 个 Agent - 模型配置和 7,308 个轨迹的评估中，得出了以下关键发现：

发现 1：技能提供了实质性但可变的收益

技能在 7 个模型 - 框架配置中平均提高性能 16.2 个百分点
但配置间差异很大（范围：+13.6pp 到 + 23.3pp）
这种可变性表明技能效果强烈依赖于特定的 Agent - 模型组合
发现 2：最佳性能配置

表现最佳的配置是 Gemini CLI 与 Gemini 3 Flash，使用技能时达到 48.7% 的通过率
值得注意的是，使用 Opus 4.5 的 Claude Code 实现了最高的改进（+23.3pp）
发现 3：自生成技能提供的收益微乎其微或为负

当被提示在解决任务前生成自己的程序知识时，模型平均比无技能基线低 1.3pp
只有 Opus 4.6 显示出适度的改进（+1.4pp）
Codex + GPT-5.2 大幅下降（-5.6pp）
其余模型持平或为负
8.2 技能效果的领域差异
技能的效果在不同领域之间差异很大：

医疗保健：+51.9pp（最高）
软件工程：+4.5pp（最低）
84 个任务中有 16 个显示负增量
这表明技能的有效性不是普遍的，而是依赖于上下文，这激励了配对评估作为 Agent 增强研究的标准实践。

8.3 技能设计原则的影响

研究还发现了以下重要结论：

少即是多：具有 2-3 个模块的专注技能优于全面的文档
规模替代效应：技能可以部分替代模型规模，使较小的模型能够在程序任务上匹配较大的模型

十、核心价值总结
10.1 从 Prompt 博弈到能力工程的转变

Agent Skills 代表了从传统的Prompt 博弈向能力工程的重要转变。通过将专家知识工程化、模块化、可复用，组织可以更高效地构建和维护专业 Agent 能力。

10.2 一次封装、多场景复用的优势

Agent Skills 实现了 "一次封装、多场景复用" 的目标，具有以下核心价值：

统一维护：所有技能都采用标准化的 SKILL.md 格式，便于统一管理和维护
降低成本：避免了为每个场景单独定制 Agent 的高昂成本
保证一致性：确保在不同场景中使用相同的专业知识和流程
10.3 通用到专用的灵活转换

Agent Skills 使通用 Agent 能够快速转变为专用 Agent，满足真实世界复杂任务的需求。这种灵活性体现在：

按需加载：只在需要时加载相关技能，保持系统轻量级
动态组合：可以同时使用多个技能，处理复杂的多领域任务
持续进化：通过不断添加和改进技能，Agent 的能力可以持续提升
10.4 开放生态系统的构建

作为一个开放标准，Agent Skills 正在构建一个繁荣的生态系统。目前已从三个来源收集了大量技能：

包括 Smithery.ai 和skillmp.com在内的社区市场：28,412 个

这种开放生态系统的构建，为 Agent Skills 的广泛应用和持续改进提供了强大动力，也为用户提供了丰富的技能资源库。
